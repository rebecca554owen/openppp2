# 安全模型

[English Version](SECURITY.md)

## 文档范围

本文从防御性工程视角描述项目：信任边界、本地执行点、攻击面和部署纪律。

## 安全目标

从代码结构看，工程试图提供以下能力：

- 受保护的隧道传输
- 显式的会话标识
- 在本地执行会话有效期、流量额度和带宽策略
- 在运维侧可控的路由和 DNS 分流
- 受控的反向访问，而不是无约束暴露
- 在保持协议统一的前提下完成平台特化的网卡与路由集成

## 信任边界

关键边界包括：

- 客户端主机
- 服务端主机
- 两者之间的承载网络
- 可选管理后端
- 本机操作系统网络栈

不要把它们视为同一个信任域。只有明确配置各边界，该设计才会稳定工作。

## 主要执行点

### 1. 传输握手与密钥状态

主要围绕 `ITransmission` 实现。

这是接纳会话并建立受保护状态的第一道关口。

### 2. 会话策略对象

主要围绕 `VirtualEthernetInformation` 实现。

其中携带：

- 带宽限制
- 剩余流量
- 到期时间

### 3. 服务端交换机与防火墙集成

主要围绕 `VirtualEthernetSwitcher` 和相关防火墙代码实现。

这里决定一个已连接会话能做什么、不能做什么。

### 4. 客户端路由与 DNS 控制

主要围绕 `VEthernetNetworkSwitcher` 实现。

这里决定哪些流量进入覆盖网络，哪些流量留在本地网络。

### 5. 后端认证与记账

启用时主要围绕 `VirtualEthernetManagedServer` 实现。

它允许外部策略系统参与，但即使后端链路不稳定，数据面也应尽量平稳退化。

## 攻击面

暴露面包括：

- TCP/UDP/WS/WSS 监听口
- 证书与私钥文件
- 本地代理监听口
- 反向映射
- 路由文件与 DNS 规则文件
- 后端凭据与 URL
- 平台特化的网络操作辅助逻辑

每多启用一个功能，暴露面就会增加。应把可选特性当作真实攻击面，而不是无害附加项。

## 安全部署建议

- 只启用真正需要的传输方式
- 当部署本来就有受管 TLS 边缘层时，优先考虑 WSS
- 证书文件和 backend key 不要进入公开仓库
- 代理凭据和数据库凭据不要保留在共享样例配置中
- 只有在明确需要对外发布服务时才开启 mappings
- 分流规则只围绕明确业务需求配置
- 路由文件和 DNS 规则应按策略资产审计

## 会话完整性与可用性

代码对以下方面较为重视：

- 超时
- 重连行为
- 保活
- 显式会话状态
- 统计与计量快照

这些同样属于安全问题，因为不稳定或模糊的会话状态会先变成运维风险，然后才会变成纯密码学问题。

## 关于混淆与格式化能力的防御视角

部分传输格式化选项用于控制兼容性与流量形态。这些能力应理解为“隧道格式控制”，而不是替代安全架构本身。

真正的安全姿态仍然依赖：

- 合理的密钥与证书
- 正确的监听暴露范围
- 正确的路由与 DNS 策略
- 受控的映射配置
- 后端凭据卫生

## 密钥管理

以下内容应按敏感信息管理：

- `protocol-key`
- `transport-key`
- `server.backend-key`
- `client.server-proxy` 中的凭据
- `websocket.ssl.certificate-key-password`
- Go 后端数据库凭据
- 启用时的 Redis 凭据

真实生产值不要保留在仓库跟踪的样例配置中。

## 加固检查清单

- 核对实际启用的监听端口
- 核对实际启用的 mappings
- 核对当前部署到底是分流还是全隧道
- 上线前核对路由文件和 DNS 规则文件
- 核对 WSS 证书链与私钥权限
- 核对后端 URL 与令牌
- 平台相关路由 / 防火墙变更先在预发布环境验证

## 安全理念

当 OPENPPP2 被当作一个可预测的网络系统来运维时，它的安全性最强：

- 拓扑清晰
- 策略清晰
- 信任边界清晰
- 启用面尽量小
- 每个目标平台都做显式验证

这才是基础设施软件应有的姿态。这里的安全核心是纪律化系统设计，而不是功能堆叠。
