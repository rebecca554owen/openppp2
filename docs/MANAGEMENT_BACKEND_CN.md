# 管理后端

[English Version](MANAGEMENT_BACKEND.md)

## 角色定位

`go/` 目录下的 Go 服务是 OPENPPP2 的管理与持久化侧，而不是分组数据面。

它为 C++ 服务端运行时提供：

- 节点认证
- 用户查询
- 额度与过期状态
- 流量记账
- HTTP 管理接口
- Redis 与 MySQL 持久化

## 为什么后端要独立存在

从代码分工看，这种拆分非常清楚：

- C++ 负责网卡、路由、socket、隧道会话和分组转发
- Go 负责业务状态、存储和管理接口

这是一种健康的基础设施拆分。数据面代码和重持久化控制面代码的演化驱动力本来就不同。

## 连接模型

Go 后端同时暴露 WebSocket 端点和 HTTP 接口。

C++ 服务端通过 `VirtualEthernetManagedServer` 接入 WebSocket 端。

这条控制链路负责：

- 后端连接握手
- echo / 健康检查
- 会话认证
- 流量上报

## 线协议格式

从代码看，后端协议不是简单 JSON 文本流，而是：

- 前 8 个十六进制字符表示 JSON 载荷长度
- 后面跟一个 JSON 对象

JSON 中包含字段：

- `Id`
- `Node`
- `Guid`
- `Cmd`
- `Data`

## 主要命令

从代码中可见的命令值包括：

- `1000`：echo
- `1001`：connect
- `1002`：authentication
- `1003`：traffic

这是一套很小但很明确的管理协议，用于 C++ 节点和 Go 后端之间协同。

## HTTP API 的作用

Go 后端还提供一组 HTTP 管理接口，用于执行管理动作，例如：

- 创建或修改用户状态
- 加载或重载用户记录
- 查询服务端记录
- 重载服务端记录

这样一来，运维和配置工作就不会堆进分组转发引擎本体。

## 持久化模型

Go 侧使用：

- Redis 作为分布式/缓存式状态层
- 通过 GORM 访问 MySQL 作为持久状态层

这再次强化了系统拆分：C++ 进程专注转发，Go 服务专注业务状态和存储。

## 为什么这对阅读 C++ 代码很重要

因为它能解释一个现象：本地运行时中的某些策略对象在后端返回前并不完整。

服务端运行时是按“可以接入外部策略系统”来设计的，但它依然保留了足够的本地结构，使自身仍然是一个完整网络节点。

## 运维意义

如果不部署后端，OPENPPP2 仍然可以以较简化的本地模式运行。

如果部署后端，则可以获得：

- 集中式认证
- 集中式流量记账
- 集中式节点与用户管理

这使得系统既适合单点基础设施部署，也适合受管服务型部署。
