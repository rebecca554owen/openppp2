# 工程哲学

[English Version](ENGINEERING_PHILOSOPHY.md)

## 项目定位

OPENPPP2 更接近网络基础设施，而不是消费级 VPN 应用。

从代码结构来看，这意味着：

- 一个运行时同时负责传输、会话策略、路由控制和虚拟网卡集成
- 系统默认运维者理解拓扑和策略
- 设计重点放在状态管理、生存能力和显式控制，而不是隐藏复杂性

## 这个项目优先优化什么

### 1. 显式网络状态

代码反复把路由、DNS、会话、IPv6、映射、监听状态显式地放进对象和配置中。

典型例子：

- `AppConfiguration`
- `VirtualEthernetInformation`
- `VirtualEthernetInformationExtensions`
- `VirtualEthernetSwitcher`
- `VEthernetNetworkSwitcher`

这种写法更像路由器软件，而不是简单的应用层隧道。

### 2. 本地自治

运行时倾向于在本地执行策略。

代码中可见的表现包括：

- 会话有效期在 C++ 进程内判断
- 流量统计在本地维护
- 路由与 DNS 策略在本地执行
- 即使没有管理后端，服务端也可以本地启动会话信息

这符合基础设施要求：即使外部控制器暂时不可用，转发进程也不应立刻丧失基本能力。

### 3. 确定性的生命周期所有权

多数核心对象都有清晰职责：

- `PppApplication`：进程生命周期
- `ITransmission`：连接握手与受保护 I/O 生命周期
- `VEthernetExchanger` / `VirtualEthernetExchanger`：会话生命周期
- `VEthernetNetworkSwitcher` / `VirtualEthernetSwitcher`：环境与系统集成生命周期

这是一种有意识的编码风格，避免把生命周期散落在过多零碎辅助类里。

### 4. 客户端与服务端共享同一协议核心

客户端和服务端共享 `VirtualEthernetLinklayer` 这套动作语义。

这样可以减少概念割裂。工程不是为每个功能单独设计一套迷你协议，而是在同一隧道动作模型上扩展能力。

### 5. 只在主机系统确有必要时做平台分化

系统并没有强行把 Windows、Linux、macOS、Android 的内部实现写成完全一样。

它选择：

- 协议逻辑统一放在 `ppp/`
- 网卡、路由、系统集成特化放在 `windows/`、`linux/`、`darwin/`、`android/`

这是一种务实的基础设施设计方法。

## 为什么这个系统会复杂

这个工程复杂，是因为它试图在一个运行时里同时解决多类问题：

- 受保护的隧道传输
- 虚拟以太网转发
- 路由与 DNS 分流
- 反向服务暴露
- 静态 UDP 路径
- 多路复用子通道
- IPv6 分配与执行
- 平台相关的网络集成

任何把它描述成“小型 VPN 客户端”的文档，都会误导读者对真实设计规模的判断。

## 为什么易用性不是第一目标

从代码看，项目面向的是愿意管理以下内容的运维者：

- 地址与掩码
- 网关与路由表
- DNS 策略
- 映射与反向暴露
- 传输类型与证书
- 平台特化运行行为

这与基础设施软件的气质一致。路由器、防火墙、覆盖网络网关的价值通常不在于“简单”，而在于“显式”和“可控”。

## 隧道设计哲学

工程没有把隧道当成一个黑盒 socket，而是拆成：

- 承载传输层
- 受保护的传输与握手层
- 隧道动作协议层
- 平台 I/O 与路由行为层

正是这种分层，才让同一运行时可以同时支持：

- TCP 和 WebSocket 承载
- TUN/TAP 集成
- FRP 风格反向映射
- 静态 UDP 分组模式
- MUX 连接复用

## 控制面哲学

OPENPPP2 把控制尽量贴近数据面。

具体表现：

- 管理后端是可选能力，而不是硬依赖
- 服务端自身维护会话表和 IPv6 租约状态
- 客户端自身维护路由与 DNS 分流决策

这对基础设施软件很重要。控制面可以外接，但本地节点仍然必须像一个完整网络系统一样工作。

## 防御哲学

从代码看，项目的实际安全重心更多是纪律化状态管理，而不是宣传话术。

关键防御特征包括：

- 显式握手
- 超时驱动的清理
- 显式会话标识
- 本地校验策略与到期状态
- 显式防火墙与路由检查
- 通过配置显式启停功能

对于想做成网络基础设施的系统来说，这才是正确的重心。

## 用什么心态阅读代码

不要把 OPENPPP2 当成一个单一算法来读。

应该把它当成一组协作子系统：

1. 进程启动与生命周期
2. 配置整形
3. 受保护的传输与帧化
4. 隧道动作协议
5. 客户端网络环境
6. 服务端会话交换机
7. 平台特化网卡与路由集成
8. 可选管理后端

后续文档都是沿着这个心智模型展开的。
