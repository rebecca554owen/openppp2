# 会话与控制面模型

[English Version](TRANSMISSION_PACK_SESSIONID.md)

## 文档目的

旧文件名沿用了 `PACK_SESSIONID` 的称呼，但真正有工程价值的话题更宽：OPENPPP2 如何标识会话、交换隧道元信息，并在传输建好之后驱动控制动作。

本文聚焦这一控制面模型。

## 核心对象

相关主要类型包括：

- `ppp/transmissions/ITransmission.*`
- `ppp/app/protocol/VirtualEthernetInformation.*`
- `ppp/app/protocol/VirtualEthernetLinklayer.*`
- `ppp/app/server/VirtualEthernetSwitcher.*`
- `ppp/app/client/VEthernetExchanger.*`
- `ppp/app/server/VirtualEthernetManagedServer.*`

## 会话标识

会话标识以 `Int128` 为中心，在客户端、服务端和管理面流程中贯穿使用。

该标识用于：

- 将一个逻辑隧道交换绑定到一个传输会话
- 在服务端定位对应 exchanger
- 关联流量计费与认证状态
- 管理 IPv6 请求、租约与控制回调

使用宽位标识符合工程目标：长生命周期会话在同一部署域内应尽可能全局唯一。

## 信息交换对象

`VirtualEthernetInformation` 承载会话级的额度与有效性信息：

- `BandwidthQoS`
- `IncomingTraffic`
- `OutgoingTraffic`
- `ExpiredTime`

这是一种最小但足够的会话策略单元，用于告诉客户端还能做什么、还能持续多久。

它与原始包转发逻辑分离，使策略交换本身保持轻量。

## IPv6 扩展

`VirtualEthernetInformationExtensions` 在基础信息对象之上补充 IPv6 状态：

- 分配的 IPv6 模式
- 分配的 IPv6 地址与前缀长度
- 网关与路由前缀
- DNS 服务器
- 请求/响应状态与状态消息

这样同一套控制消息族即可同时承载通用会话策略与 IPv6 下发结果。

## 握手后的控制动作

传输握手完成后，链路层可以继续交换多种控制动作：

- 信息同步
- 保活
- TCP connect/push/disconnect
- UDP sendto
- echo / echo reply
- 静态路径建立
- mux 建立
- FRP 风格映射注册与数据转发

关键点在于，OPENPPP2 并没有把这些当成彼此无关的子系统，而是统一建模为同一隧道控制面中的动作。

## 客户端流程

高层流程如下：

1. 构建配置与本地网络上下文
2. 打开虚拟网卡并准备路由策略
3. 创建 `VEthernetExchanger`
4. 向远端服务端建立传输会话
5. 完成握手并取得会话标识
6. 交换 `VirtualEthernetInformation`
7. 应用路由、DNS、mux、代理、映射和可选 IPv6 状态
8. 进入稳态包转发与保活循环

`VEthernetExchanger` 是传输会话与客户端虚拟网络状态之间的运行桥梁。

## 服务端流程

高层流程如下：

1. 为启用的传输类型打开监听器
2. 接收新的传输连接
3. 完成服务端握手
4. 创建或附着对应的 `VirtualEthernetExchanger`
5. 接纳会话并构造信息信封
6. 在启用时通过管理后端校验或补充状态
7. 维护隧道流量、IPv6 租约、NAT 状态、映射和统计信息

`VirtualEthernetSwitcher` 扮演会话交换机与生命周期协调器的角色。

## 管理面的角色

`VirtualEthernetManagedServer` 是可选组件。它通过 WebSocket 或安全 WebSocket 把隧道服务端接入外部控制系统。

其职责包括：

- 异步认证
- 流量上报与统计
- 后端可达性检查
- 管理链路重连

这样数据面仍留在 C++ 进程内，而策略、计费、用户管理则可以位于外部系统。

## 额度、过期与准入

会话模型内置了以下显式约束点：

- 流量额度
- 到期时间
- 带宽 QoS 限制
- 后端介入式认证

这对于 SD-WAN 和受管 VPN 场景很重要，因为即便外部后端短时变慢或不可用，隧道运行时也仍应在本地执行基本策略。

## 映射与反向访问

控制面同时承载映射状态。客户端侧通过 `client.mappings` 定义需要暴露的服务，对应的 FRP 风格动作负责：

- 注册
- 建链
- 数据推送
- 断开
- UDP sendto 中继

因此该覆盖网络不仅能做远程接入，也能做受控的反向暴露通道。

## Static Echo 路径

客户端和服务端 exchanger 中都包含 static echo 机制，其意义是运行层面的，而不是装饰性的：

- 维持静态 UDP 风格路径的活性
- 验证可达性
- 维持数据报型会话状态

它属于控制面，因为它表达的是会话健康状态，而不是普通业务负载。

## 失败与恢复模型

会话设计默认会发生失败，并内置了明确恢复钩子：

- 握手超时
- 重连超时
- 传输释放与清理
- 基于保活的健康检测
- 客户端 exchanger 的状态迁移
- 服务端管理链路的重连行为

这符合网络基础设施对自治性和低运维惊扰的要求。

## 设计理念

该控制面体现了几条稳定原则：

- 身份、策略、包转发相互关联，但不混杂实现
- 会话状态应体现在类型中，而不是散落在零碎 socket 代码里
- 一个 exchange 对象应尽量负责一条逻辑客户端/服务端关系的生命周期
- 外部管理应是可选能力，而不是让数据面强依赖它

## Session ID 打包机制详解

### 核心打包函数架构

`Transmission_Handshake_Pack_SessionId` 是会话标识序列化的核心入口，其实现位于传输层 handshake 模块中。该函数接受一个 `Int128` 类型的 session_id 值作为输入，经过多层变换后输出一个看似随机的字符串 payload。与普通整数序列化不同，这里输出的并非十进制数字的明文，而是经过精心设计的混淆字节序列。

函数的实现路径根据输入值分成两条截然不同的分支：当 session_id 为零时进入 dummy 包路径，当 session_id 非零时进入真实包路径。这种设计使得同一套打包逻辑能够同时服务于两类语义完全不同的场景——既可以生成承载真实会话标识的有效载荷，也可以生成仅用于制造流量噪声的虚假数据包。

### 首字节编码规则

首字节的构造是整个打包流程中最关键的决策点，因为它同时承担了两项职责：标识数据类型和携带混淆密钥的部分材料。

对于真实包路径，首字节的取值范围被限制在 `0x00` 到 `0x7f` 之间，这保证了最高位为 0。接收端在解包时通过检查首字节的最高位来判断当前处理的是真实包还是 dummy 包——最高位为 0 意味着后续内容是可解析的真实会话标识。

对于 dummy 包路径，首字节的取值范围在 `0x80` 到 `0xff` 之间，此时最高位为 1。接收端识别到这一模式后，会将整个包标记为可丢弃的扰动项，而不会尝试将其解析为有效的会话标识。这种首字节编码策略使得真实包和 dummy 包在二进制层面呈现出明显的统计差异，但这种差异被有意引入的随机填充所掩盖。

### 核心数值处理

真实包路径中，session_id 的 Int128 整数值首先被转换为十进制字符串形式。例如，一个典型的 session_id 可能被转换为类似 "28014476829478429384729834729384" 这样的字符串。这个十进制字符串构成了 payload 的核心内容，后续的所有填充和变换都围绕这个核心展开。

在 dummy 包路径中，由于 session_id 为零，代码会生成一个随机的 Int128 值来替代零值。这个随机值的生成使用了 GUID 风格的算法，确保每次生成的 dummy 包内容都不相同。这样做的好处是，即使攻击者能够观察到大量 dummy 包，也无法从中推断出任何有意义的模式，因为每次的内容都是独立随机生成的。

### 追加字节序列

在核心数值处理完成后，打包流程会继续追加一系列辅助字节。这些追加项并非简单的填充，而是与密钥材料紧密相关：

首先追加三个随机非零字节。这三个字节的取值范围排除了零值，使得 payload 的长度和内容更加不可预测。这三个字节在后续的密钥扰动阶段会发挥作用。

然后追加一个分隔字符。这个分隔符的选择受到配置的影响，其目的，是在核心数值和后续填充之间建立明确的边界，同时不引入任何可能泄露信息的可识别模式。

接下来追加受 `key.kx` 影响的随机填充。`key.kx` 是配置中的关键材料之一，它直接影响填充的长度和分布。这个填充阶段的实现确保了即使两个 session_id 相同的包，在不同的时间点发送到不同的对端，也会产生完全不同的输出。

在某些分支条件下，payload 还会追加斜杠字符 "/"，这进一步增加了输出格式的不确定性。

最后追加随机可打印字符序列。这些字符的作用是使最终的 payload 呈现出类似随机文本的外观，而不是结构化的二进制数据。

### 密钥扰动与 XOR 变换

完成上述构造后，打包流程进入最关键的混淆阶段。代码使用前面提到的四个前缀字节来逐步扰动 `kf` 变量，然后对整个 payload 执行循环 XOR 变换。

这个 XOR 变换并非简单的单次操作，而是多轮迭代。每轮迭代都会使用不同的密钥材料对 payload 的不同位置进行变换。这种设计使得逆向工程变得极为困难——攻击者无法通过简单的线性分析来还原原始的 session_id 值。

变换过程中使用的密钥材料来自于多个来源的混合：`kf` 的初始值、前面生成的四个前缀字节、以及 payload 自身的某些部分。这种自引用的设计使得破解难度进一步增加。

### 解包流程详解

`Transmission_Handshake_Unpack_SessionId` 执行完全相反的操作，将混淆后的 payload 还原为原始的 Int128 会话标识。

解包流程的第一步是基础长度检查。如果输入 payload 的长度低于某个最小阈值，函数会立即返回错误，因为一个有效的 session-id 包不可能如此短小。

第二步是首字节分析。解包函数读取 payload 的第一个字节，检查其最高位。如果最高位为 1，则意味着当前处理的是一个 dummy 包，函数会设置相应的标记并立即返回，通知调用者这是一个应被丢弃的包。如果最高位为 0，则提取四个前缀字节用于后续的反向变换。

第三步是逆向 XOR 恢复。使用前面提取的四个前缀字节，解包函数对 payload 执行与打包时相反的 XOR 变换，逐步还原出原始的核心数值字符串。

第四步是十进制解析。将恢复后的字符串按照十进制格式解析为 Int128 类型的整数值，这就是原始的 session_id。

### 接收端的持续读取机制

在实际的握手交互中，接收端的 `Transmission_Handshake_SessionId` 函数实现了一个持续读取循环。这个循环的设计是为了配合 NOP 前奏机制而专门优化的。

当客户端或服务端开始等待接收 session-id 类型的握手项时，它会反复调用解包函数，直到成功获取到一个非 dummy 的真实包才返回。这种设计天然支持了 NOP 前奏的工作方式——发送端会先发送若干个 dummy 包作为握手前奏的扰动流量，而接收端则会持续读取并自动跳过这些 dummy 包，直到真实的对端会话标识出现。

这个机制的设计哲学很有意思：它将流量噪声的过滤内置到了接收逻辑中，而不是要求发送端在发送真实数据之前先发送某种“开始”信号。这种设计使得整个握手过程在时序上更加自然，也更难被外部观察者识别出明确的阶段边界。

### Session ID 在 Int128层面的语义

OPENPPP2 选择 Int128 作为会话标识的底层表示，这并非随意之举。128 位的宽度提供了足够大的地址空间，使得在同一部署域内几乎不可能发生会话标识冲突。对于需要长时间运行的隧道会话，这种全局唯一性是非常重要的工程需求。

Int128 的使用还带来了一个额外的好处：它可以与 GUID 格式进行无损转换。代码中多处使用 `Int128ToGuidString` 和 `GuidStringToInt128` 来实现两种表示形式之间的互转。这种互操作性在日志输出、调试追踪、以及与外部系统交互时非常有用。

从类型系统的角度看，将 session_id 明确建模为 Int128 类型而非普通的 64 位整数或字符串，能够在编译时提供更强的类型检查，减少错误的可能性。

### 打包结果的网络表示

经过上述所有变换后，最终在网络上传输的 session-id 包呈现出以下特征：

包的长度是可变的，取决于随机填充的数量和追加的可打印字符序列。没有任何固定的字段边界可以用于识别包的结构。包的内容看起来像随机生成的文本，没有明显的模式。所有数值相关的部分都经过了 XOR 混淆，无法直接还原。

这种网络表示的设计目标是最大化流量分析的难度，使得被动监控者难以从加密前的握手流量中提取出有效的会话标识信息。

## 与握手序列的协同

Session ID 的打包机制是整个握手序列的有机组成部分。在 `InternalHandshakeServer` 中，服务端生成一个非零的 Int128 作为 session_id，然后通过 `Transmission_Handshake_Pack_SessionId` 进行打包并发送给客户端。客户端接收后使用 `Transmission_Handshake_Unpack_SessionId` 进行解包，获得真实的会话标识。

在 NOP 前奏阶段，发送端使用 session_id = 0 调用打包函数，这会自动触发 dummy 包路径。接收端在解包时检测到首字节最高位为 1，会将其标记为 dummy 并丢弃。这样，NOP 包就在不使用任何特殊标记的情况下实现了“看似有效实则可丢弃”的语义。

`ivv` 和 `nmux` 的交换也复用了同一套打包机制。客户端生成 `ivv` 时使用 GUID 派生的 Int128 值，服务端生成 `nmux` 时先创建一个随机值然后调整最低位来携带 mux 状态信息。这些值都通过相同的 pack/unpack 接口进行传输。

## 安全意义

从安全角度审视 session-id 的打包设计，可以看到几个关键的防御目标：

流量混淆方面，dummy 包的引入增加了握手流量的不确定性。每次握手的 NOP 包数量由 `key.kl` 和 `key.kh` 动态计算，这意味着即使是同一个客户端连接到同一个服务端，每次握手的噪声量也是不同的。

控制信息保护方面，XOR 变换和多层密钥扰动确保了即使攻击者能够截获大量的 session-id 包，也无法通过分析这些包的模式来推断出有效的会话标识。

密钥派生方面，session-id 本身并不参与密钥派生，真正的密钥派生输入是客户端独立生成的 `ivv`。这种设计使得会话标识的交换和密钥材料的交换是解耦的。

状态标记隐蔽方面，mux 状态被嵌入到 `nmux` 的最低位，而不是作为一个独立的控制字段发送。这使得状态信息在统计上与随机数无法区分。
