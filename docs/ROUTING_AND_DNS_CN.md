# 路由与 DNS 设计

[English Version](ROUTING_AND_DNS.md)

## 为什么这一部分很重要

在 OPENPPP2 中，路由和 DNS 行为是产品核心组成部分，而不是附带副作用。

这也是为什么代码里专门存在以下能力：

- bypass IP 列表
- route 文件
- 远端路由刷新
- DNS 规则文件
- DNS 重定向
- DNS 缓存协作
- 默认路由保护

## 路由控制模型

客户端掌握了大部分路由分流逻辑。

关键输入包括：

- `--nic`
- `--ngw`
- `--bypass`
- `--bypass-ngw`
- `client.routes`
- `--virr`
- `--tun-vnet`
- `--tun-host`

代码利用这些输入决定：

- 哪些流量进入隧道
- 哪些流量留在本地网络
- 如何避免隧道服务器本身被错误路由回覆盖网络中

## Bypass 列表

Bypass 文件为客户端提供了一种简单而显式的方式，把部分目标保持在覆盖网络之外。

这比试图从应用行为里推断分流策略更直接、更安全。

## Route 文件与 vBGP 风格列表

客户端路由项可以同时包含：

- 本地 `path`
- 远端 `vbgp` URL

这意味着客户端可以使用文件驱动的路由策略，并按周期刷新，而不必把所有路由决策都实时交给大型控制器。

## 默认路由保护

Linux 客户端显式实现了 protect 模式。更一般地说，客户端运行时会尽量避免自递归路由和意外黑洞路由。

这对基础设施非常关键。覆盖网络客户端必须避免把自己的控制流量重新送回覆盖网络路径里。

## DNS 规则模型

DNS 规则让客户端把 DNS 当作流量分类信号。

这很有价值，因为在覆盖网络系统中，很多决策是基于域名，而不仅仅是基于前缀。

因此客户端可以让 DNS 流量与普通 UDP 流量走不同路径。

## DNS 重定向

客户端 UDP 路径会检测数据报是否是 DNS 流量，并在需要时将其重定向到指定服务器。

这不只是性能优化，而是为了让名称解析与覆盖网络策略保持一致。

## 服务端 Namespace Cache

服务端内部包含一套 DNS 风格数据的 namespace cache，用来减少重复解析，并帮助隧道侧名称解析行为更加可预测。

## 为什么路由和 DNS 要放在同一篇文档里

因为在这个系统里，它们在运行上是绑定在一起的。

例如：

- bypass 路由列表决定部分目标保持本地
- DNS 规则决定某些域名应该走覆盖网络
- 路由安装过程中还可能要特殊处理 DNS 服务器本身的路径

如果把路由策略和 DNS 策略完全割裂书写，运维者就很难真正理解流量是如何被分类的。

## 基础设施视角

从路由器式视角看，路由和 DNS 控制正是覆盖网络具备策略意识的关键。

没有它们，OPENPPP2 只是一个受保护的传输层；有了它们，它才成为一个真正能够执行覆盖网络策略的边缘节点。
